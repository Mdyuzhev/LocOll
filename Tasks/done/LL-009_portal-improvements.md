> Дата создания: 2026-03-15

## Цель

Улучшить Lab Portal (`http://100.81.243.12:4000`) по результатам живого аудита
проведённого через Claude in Chrome + /claude-bridge. Портал технически здоров —
все API отвечают, Alpine/htmx/WASM загружены, SSE работает. Задача — устранить
конкретные проблемы и добавить недостающую функциональность.

**Сервер прямо сейчас:** CPU ~2%, RAM 3.7/23.9 GB, диск 143/914 GB, uptime 79 дней.

**25 контейнеров running**, из них 3 unhealthy:
`docker-automation-worker-1`, `docker-notification-worker-1`, `docker-generator-1`

**Services раздел** знает только о 5 сервисах (захардкожено в `handlers/services.go`):

по LAN (`192.168.1.74:4000`) — таймаут (это нормально, LAN не всегда доступен
с dev-машины).

## Задачи по приоритету

### 🔴 Приоритет 1 — Services: динамический список вместо хардкода

**Проблема.** `internal/handlers/services.go` содержит захардкоженный срез
`serviceTargets` из 5 элементов. При добавлении новых сервисов нужно менять код.
Три из пяти сейчас показывают "down" потому что Warehouse-стек не запущен —
это вводит в заблуждение.

**Решение.** Два уровня:

Первый — **автоматический**: добавить в Services раздел блок "Docker Containers"
который берёт данные из уже существующего `/api/v1/containers`. Группировать
по `project` label. Показывать статус (running/unhealthy/stopped) с цветовой
индикацией. Это ничего не ломает и не требует нового кода — только новый
htmx-фрагмент.

Второй — **конфигурируемый**: вынести список HTTP health-check целей из хардкода
в переменную окружения `LOCOLL_SERVICES` (JSON или просто строка с URL через
запятую). Если переменная не задана — использовать дефолтный список. Тогда
добавить новый сервис можно через `.env` без перекомпиляции.

Формат переменной:
```
LOCOLL_SERVICES=Ollama:http://localhost:11434,ErrorLens:http://localhost:8002/health,Grafana:http://localhost:3001
```

### 🔴 Приоритет 2 — Unhealthy контейнеры: видимый алерт на главной

**Проблема.** Три unhealthy контейнера видны только если перейти на вкладку
Containers и посмотреть иконки. На главной (Server вкладка) никакого сигнала нет.

**Решение.** Добавить на Server вкладку небольшой блок "Alerts" который появляется
только когда есть unhealthy или stopped контейнеры. SSE-брокер уже обновляет
`#server-metrics` раз в минуту — добавить рядом `#alerts-block` который обновляется
тем же потоком.

Формат блока:
```
⚠️ Требуют внимания (3)
docker-automation-worker-1  unhealthy  3 дня
docker-notification-worker-1  unhealthy  3 дня
docker-generator-1  unhealthy  3 дня
```

Если всё здорово — блок не рендерится вообще (не показывать пустой "✅ всё ок",
это шум).

В `collector.go` уже есть логика определения состояния контейнеров. В
`simpleRenderer.RenderServerCard` — добавить рендер alerts рядом с метриками.

### 🟡 Приоритет 3 — Метрики: диагностика и починка истории

**Проблема.** История метрик начинается только с сегодня (`2026-03-15`), хотя
портал работает 28+ часов (контейнер поднят 14 марта). Нужно разобраться почему.

**Что проверить агенту:**
```bash
# Сколько записей в БД
docker exec locoll-backend-1 sqlite3 /app/data/portal.db "SELECT COUNT(*), MIN(datetime(ts,'unixepoch')), MAX(datetime(ts,'unixepoch')) FROM metrics"

# Есть ли данные вообще
docker exec locoll-backend-1 sqlite3 /app/data/portal.db "SELECT * FROM metrics ORDER BY ts DESC LIMIT 5"

# Размер volume
docker volume inspect locoll_portal_data
```

Если данные есть в БД но `/metrics/history` отдаёт мало — проблема в параметрах
запроса (по умолчанию `hours=1` — только последний час). Исправить дефолт на
`hours=24` в `handlers/metrics.go`.

Если данных нет — значит коллектор не писал. Проверить логи:
```bash
docker logs locoll-backend-1 --tail 100 | grep -i "metric\|collect\|error"
```

### 🟡 Приоритет 4 — AI раздел: список контейнеров не подгружается

**Проблема.** В AI разделе есть `select.select-container` который должен
заполняться через `fetch('/api/v1/containers')` в `aiApp().init()`. Судя по коду
в `page.go`, это происходит при инициализации Alpine-компонента. Нужно проверить
работает ли это на практике — открыть вкладку AI и убедиться что дропдаун
заполнен контейнерами.

Если не заполняется — проверить что `x-data="aiApp()"` инициализируется когда
вкладка не активна (Alpine может не вызывать `init()` для скрытых элементов).
Исправление: перенести fetch в `x-init` с ленивой загрузкой аналогично Analytics.

### 🟢 Приоритет 5 — Notes: виджет активных заметок на главной

**Проблема.** Система заметок (`/note` в Telegram → homelab-mcp → SQLite) работает,
но портал о ней ничего не знает. Заметки видны только агенту при `start_session`.

**Решение.** Добавить на Server вкладку небольшой виджет "📝 Заметки" аналогично
alerts-блоку. Виджет запрашивает `GET /api/v1/notes` (новый эндпоинт) и показывает
список невыполненных заметок с кнопкой "✓" для отметки выполненной.

**Новый эндпоинт** `GET /api/v1/notes` — прокси к homelab-mcp `/claude-bridge`:
```go
// handlers/notes.go
func (h *Handler) Notes(w http.ResponseWriter, r *http.Request) {
    // POST к http://localhost:8765/claude-bridge
    // {"tool": "list_notes", "args": {}}
    // Отдать результат как JSON
}
```

Кнопка "✓" делает `POST /api/v1/notes/{id}/complete` → bridge → `complete_note`.

Это замыкает цикл: написал в Telegram → видно в портале → отметил выполненным.

### 🟢 Приоритет 6 — UX мелочи

Список небольших улучшений которые стоит сделать попутно:

- **Заголовок страницы**: сейчас `LocOll — Lab Portal`, добавить в `<title>`

- **Uptime в шапке**: показать uptime сервера в `<header>` рядом с логотипом —

  — сейчас непонятно насколько свежие данные
- **Вкладка по умолчанию из URL**: добавить поддержку `?tab=containers` чтобы
  можно было сделать закладку на конкретный раздел
- **`/api/v1/health` расширить**: сейчас возвращает просто `{"status":"ok"}`.
  Добавить `uptime`, `containers_total`, `containers_unhealthy`, `db_size_mb` —
  полезно для мониторинга и для агента
