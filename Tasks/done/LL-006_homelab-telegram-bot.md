# LL-006: homelab-telegram-bot — Telegram-бот для управления и мониторинга сервера

> Статус: Готово к разработке  
> Дата создания: 2026-03-13

---

## Цель

Создать Telegram-бота который живёт на сервере и является мобильным интерфейсом
к homelab-инфраструктуре. Бот решает одну практическую проблему: **видеть что
происходит с сервером и реагировать на инциденты без ноутбука** — с телефона,
из метро, в любое время.

Бот — не замена порталу LocOll. Это дополнение: быстрые команды, алерты в реальном
времени и AI-объяснения через Ollama для тех случаев когда открывать браузер
неудобно.

---

## Техническая среда

Бот живёт на сервере `192.168.1.74` как Docker-контейнер в составе нового
Compose-стека `homelab-bot`. Данные берёт из существующих сервисов через localhost:
homelab-mcp на порту 8765, LocOll portal API на порту 8010, Ollama на порту 11434.
Никаких новых сетевых зависимостей — только внутренние порты сервера.

Стек: **Python 3.11**, библиотека `python-telegram-bot` v21+ (async). Транспорт
с Telegram: **polling** — бот сам опрашивает Telegram API каждые 2 секунды. Webhook
не подходит: у сервера нет публичного IP.

Конфигурация через переменные окружения в `.env` файле:

```
TELEGRAM_BOT_TOKEN=<токен от @BotFather>
ALLOWED_CHAT_IDS=<твой chat_id, через запятую если несколько>
HOMELAB_MCP_URL=http://localhost:8765
LOCOLL_API_URL=http://localhost:8010
OLLAMA_URL=http://localhost:11434
OLLAMA_MODEL=mistral
```

## Авторизация

Это критически важно — бот торчит в интернете через Telegram API. Каждый входящий
`Update` (сообщение, callback) проходит через `auth.py` до того как попасть в
хендлер. Логика простая: взять `update.effective_chat.id`, сравнить со списком
`ALLOWED_CHAT_IDS`. Если не совпадает — ответить `"⛔ Доступ запрещён"` и
залогировать попытку в stderr. Никаких исключений, никаких обходов.

Как узнать свой `chat_id`: написать боту любое сообщение, он залогирует chat_id
в stderr при первом обращении от незнакомого пользователя.

## Команды бота

### `/status` — обзор сервера

Запрашивает `GET /api/v1/system` и `GET /api/v1/containers` у LocOll portal.
Возвращает компактную сводку:

```
🖥 flomasterserver

💾 RAM: 4.8 / 23 GB (21%)
⚡ CPU: 23% | Load: 0.45
💿 Disk: 115 / 915 GB (12%)
⏱ Uptime: 14 дней

📦 Контейнеры: 20 running / 21 total
⚠️ Unhealthy: warehouse-analytics, docker-notification-worker-1

🕐 Обновлено: 14:32:05
```

Строка `⚠️ Unhealthy` показывается только если есть unhealthy контейнеры. Если
всё зелёное — строка `✅ Все контейнеры здоровы`.

---

### `/containers [проект]` — список контейнеров

Без аргумента — показывает все проекты и количество контейнеров в каждом.
С аргументом (например `/containers locoll`) — показывает контейнеры конкретного
проекта со статусами.

```
📦 LocOll

locoll-frontend  ✅ running  6 часов
```

Проект определяется по Docker Compose label `com.docker.compose.project` — данные
берутся из LocOll API `/api/v1/containers`.

---

### `/logs <имя_контейнера> [строк]` — логи контейнера

Запрашивает последние N строк логов через LocOll API `/api/v1/containers/{id}/logs`.
По умолчанию 20 строк. Максимум 50 — иначе сообщение не влезет в Telegram.

Пример: `/logs locoll-frontend 30`

Если контейнер не найден — ответить `"❌ Контейнер 'xxx' не найден. Используй
/containers чтобы посмотреть список."`.

Длинный вывод разбить на несколько сообщений по 4096 символов (лимит Telegram).

---

### `/restart <имя_контейнера>` — перезапуск контейнера

**Требует подтверждения.** После команды бот отвечает inline-кнопками:

```
Перезапустить locoll-frontend?
[✅ Да, перезапустить]  [❌ Отмена]
```

При нажатии "Да" — POST запрос к LocOll API `/api/v1/containers/{id}/restart`.
При нажатии "Отмена" — `"Отменено."`. Подтверждение истекает через 60 секунд —
после этого кнопки становятся неактивными.

Это единственная команда которая *изменяет* состояние сервера. Поэтому она требует
явного подтверждения.

---

### `/ask <вопрос>` — AI-вопрос через Ollama

Агрегирует контекст (текущий статус системы + последние события из homelab-mcp)
и отправляет в Ollama с вопросом пользователя.

Пример: `/ask почему locoll-frontend перезапускался сегодня ночью?`

Промпт к Ollama строится так:

```
Ты — ассистент по управлению домашней лабораторией.
Отвечай на русском языке кратко и по делу.

Текущее состояние сервера:
<данные из /api/v1/system>

События контейнеров за последние 24 часа:
<данные из homelab-mcp /api/v1/events>

Вопрос пользователя: <вопрос>
```

Модель: `mistral` (уже загружена). Тайм-аут запроса к Ollama: 30 секунд. Пока
Ollama думает — отправлять промежуточное сообщение `"🤔 Думаю..."` и потом
редактировать его ответом. Если тайм-аут — `"❌ Ollama не ответила за 30 секунд.
Попробуй позже."`.

## Алерты (фоновый мониторинг)

Это главная ценность бота для режима "я не за ноутбуком". Фоновая задача `watcher.py`
запускается при старте бота как `asyncio.create_task` и работает в бесконечном цикле
с интервалом **60 секунд**.

На каждой итерации `watcher.py` запрашивает `GET /api/v1/containers` у LocOll portal
и сравнивает с предыдущим состоянием (хранится в памяти процесса как dict
`{container_name: status}`). Правила отправки алерта (`rules.py`):

Алерт отправляется когда контейнер **перешёл** в новое состояние, а не просто
находится в нём. Это важно — иначе бот будет спамить каждые 60 секунд про
warehouse-analytics который unhealthy уже 4 дня.

| Переход | Алерт |
|---------|-------|
| running → unhealthy | `⚠️ <контейнер> стал unhealthy` |
| running → exited | `🔴 <контейнер> остановился` |
| * → running (после не-running) | `✅ <контейнер> восстановился` |
| новый контейнер появился | `🆕 <контейнер> запущен` |
| контейнер исчез из списка | `🗑 <контейнер> удалён` |

Дополнительные пороговые алерты — проверяются каждые **5 минут** (каждая 5-я
итерация):

RAM > 85% → `⚠️ RAM: X GB / 23 GB (Y%)`
Disk > 80% → `⚠️ Диск: X GB / 915 GB (Y%)`

Алерты уходят **всем** chat_id из `ALLOWED_CHAT_IDS`.

При старте бота watcher инициализирует baseline-состояние без отправки алертов —
чтобы не спамить про уже существующие unhealthy контейнеры.

## Docker и деплой

### `Dockerfile`

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY bot/ ./bot/
CMD ["python", "-m", "bot.main"]
```

### `docker-compose.yml`

```yaml
services:
  homelab-bot:
    build: .
    network_mode: host        # чтобы видеть localhost:8765, :8010, :11434
    env_file: .env
    restart: unless-stopped
    volumes:
      - /var/log/homelab-bot:/app/logs
```

`network_mode: host` — та же причина что у LocOll backend: нужно видеть
localhost-порты сервера. Наружу ничего не торчит — бот только сам ходит в Telegram.

### Деплой

Агент создаёт директорию `~/projects/homelab-bot/` на сервере через paramiko SFTP,
передаёт все файлы, создаёт `.env` из шаблона с реальными значениями переменных
(токен подставить из задачи или запросить у пользователя), затем выполняет
`docker compose up -d --build`.

**Перед деплоем:** убедиться что LocOll API отвечает на `localhost:8010/api/v1/health`
и homelab-mcp отвечает на `localhost:8765`. Если не отвечают — остановиться и
сообщить.

## Порядок выполнения

Сначала проверить доступность зависимостей через paramiko: `curl -s localhost:8010/api/v1/health`
и `curl -s localhost:8765/api/v1/system`. Посмотреть реальные форматы ответов — писать
клиентов под реальность, а не под предположения.

Затем создать структуру файлов, начать с `auth.py` и `clients/` — это фундамент.
Потом команды в порядке приоритета: `/status` → `/containers` → `/logs` →
алерт-воркер → `/restart` → `/ask`.

Последний шаг — деплой через paramiko и проверка критериев готовности.

## Критерии готовности

| Проверка | Как убедиться |
|---|---|
| Бот отвечает | Написать `/status` в Telegram — получить сводку сервера |
| Авторизация работает | Написать с другого аккаунта — получить `⛔ Доступ запрещён` |
| `/containers locoll` | Показывает контейнеры LocOll со статусами |
| `/logs locoll-frontend` | Показывает последние 20 строк логов |
| `/restart` требует подтверждения | Inline-кнопки появляются, действие выполняется только после "Да" |
| Алерт при падении | Остановить тестовый контейнер — прийти алерт в Telegram |
| Алерт при восстановлении | Поднять контейнер обратно — прийти алерт `✅ восстановился` |
| `/ask` отвечает | Задать вопрос — получить ответ от Ollama на русском |
| Бот выживает после рестарта | `docker compose restart homelab-bot` — бот снова работает |
| Нет спама при старте | При первом запуске алерты про уже-unhealthy контейнеры не приходят |

## Запрещено

- `ssh`, `sshpass` — только paramiko для деплоя
- Webhook — только polling
- Хардкодить токен или chat_id в коде — только через `.env`
- Отвечать на команды от неавторизованных chat_id — только `⛔ Доступ запрещён`
- Слать алерт если контейнер уже был в этом состоянии при старте бота
- Тянуть новые модели Ollama — только `mistral` который уже загружен
- Разрешать `/restart` без подтверждения inline-кнопками
