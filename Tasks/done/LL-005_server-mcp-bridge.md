# LL-005: server-mcp-bridge — связь локального agent-context с серверным MCP

> Статус: ✅ Выполнена
> Дата создания: 2026-03-13
> Дата завершения: 2026-03-13
> Зависимость: LL-004 (agent-context локальный сервер) — выполнена

---

## Цель

Добавить в локальный `agent-context` сервер модуль `server-bridge.js` который при
старте каждой сессии запрашивает серверный MCP (`homelab-mcp`, порт 8765 по
Tailscale) и обогащает контекст агента реальными данными о состоянии production
среды. Агент получает не просто "что мы делали в прошлый раз", а полную картину:
что происходило на сервере пока разработчик не работал.

Связь односторонняя: локальный → серверный (pull). Серверный MCP не знает о
существовании локального — он просто отвечает на запросы. Если сервер недоступен
— локальный сервер работает без моста, graceful degradation.

---

## Техническая среда

Серверный MCP работает на `192.168.1.74:8765` (LAN) / `100.81.243.12:8765`
(Tailscale), транспорт SSE. Это уже существующий сервис, **его не трогаем**.

Локальный agent-context живёт в `~/.agent-context/` на Windows-машине разработчика.

Транспорт для связи: HTTP запросы от `server-bridge.js` к серверному MCP через
Tailscale. Адрес настраивается в `registry.json` в секции `global` — поле
`homelab_mcp_url`. По умолчанию `http://100.81.243.12:8765`. Если поле отсутствует
— мост отключён.

## Что добавляется

### Новый файл: `~/.agent-context/server-bridge.js`

Модуль с одной публичной функцией `fetchServerDigest(projectName, sinceTimestamp)`.
Функция делает HTTP запрос к серверному MCP и возвращает структурированный дайджест
событий. Вызывается из `start_session` и `get_context` после определения активного
проекта.

Запрос к серверному MCP — это HTTP POST на `/api/v1/events` с телом:

```json
{
  "project": "locoll",
  "since": "2026-03-11T14:00:00Z"
}
```

Параметр `since` — дата окончания последней сессии из локальной БД. Если
предыдущих сессий нет — брать `now() - 24h`.

Ответ от серверного MCP — массив событий:

```json
[
  {
    "ts": 1741700000,
    "container": "locoll-backend-1",
    "event": "restarted",
    "detail": "exit code 1, OOM"
  }
]
```

Тайм-аут запроса: **3 секунды**. При любой ошибке (сеть, тайм-аут, невалидный JSON)
— логировать в stderr и возвращать `null`. Вызывающий код обязан проверять `null`
и продолжать работу без дайджеста.

---

### Изменения в `tools/start_session.js`

После сборки основного контекста (конвенции + последняя сессия) — вызвать
`fetchServerDigest`. Если результат не `null` и содержит события — добавить в
ответ агенту новую секцию:

```
[Сервер с момента последней сессии]
locoll-backend-1: restarted (exit code 1, OOM) — 2 дня назад
locoll-backend-1: healthy — 2 дня назад
⚠️ 1 перезапуск контейнера требует внимания
```

Формат каждой строки: `<container>: <event> [— <detail>] — <время_назад>`.
Время считать от `ts` до `now()` и форматировать как "2 дня назад", "3 часа назад",
"45 минут назад". Строка с `⚠️` добавляется если есть события типа `restarted`,
`unhealthy`, `stopped` — т.е. события требующие внимания. `started` и `healthy`
события считаются информационными.

Если событий нет совсем — добавить одну строку:
`[Сервер: инцидентов не было с момента последней сессии]`

Если мост вернул `null` — секцию не добавлять вообще, молча пропустить.

---

### Изменения в `tools/get_context.js`

Аналогично `start_session` — та же секция с дайджестом. Параметр `since` — дата
окончания последней **завершённой** сессии (не текущей активной).

---

### Изменения в `registry.json`

Добавить в секцию `global` два поля:

```json
"homelab_mcp_url": "http://100.81.243.12:8765",
"homelab_mcp_enabled": true
```

Если `homelab_mcp_enabled` равно `false` или поле отсутствует — `server-bridge.js`
немедленно возвращает `null` без попытки подключения.

## Новый tool: `server_status`

Добавить шестой инструмент (итого 7 tools) для явного запроса состояния сервера
без привязки к сессии. Полезен когда агент хочет проверить сервер в любой момент,
не только при старте.

Параметры: нет. Не требует активной сессии.

Логика: вызвать `fetchServerDigest` с `since = now() - 1h` и дополнительно
запросить у серверного MCP текущие метрики через `GET /api/v1/system` и
`GET /api/v1/containers`. Показать агенту полный срез: метрики железа + статусы
контейнеров проекта + события за последний час.

Формат ответа:

```
=== Статус сервера: 192.168.1.74 ===

[Железо]
CPU: 23% | RAM: 4.8 / 23 GB (21%) | Disk: 115 / 915 GB (12%)
Load: 0.45 / 0.38 / 0.31 | Uptime: 14 дней

[Контейнеры LocOll]
locoll-backend-1  ✅ running  healthy   3 дня
locoll-nginx-1    ✅ running  —         3 дня

[События за последний час]
Инцидентов не было
```

Если серверный MCP недоступен — возвращать:
`"❌ Серверный MCP недоступен (100.81.243.12:8765). Проверь Tailscale."`

## Порядок выполнения

Сначала убедиться что серверный MCP отвечает: выполнить `curl -s
http://100.81.243.12:8765/api/v1/system` через paramiko и проверить ответ. Это
важно сделать до написания кода — нужно знать реальный формат ответа серверного MCP,
а не предполагать его.

Затем написать `server-bridge.js` с функцией `fetchServerDigest` и отдельно
протестировать её изолированно — просто запустить `node -e
"import('./server-bridge.js').then(m => m.fetchServerDigest('locoll',
'2026-03-01T00:00:00Z').then(console.log))"`.

После этого интегрировать в `start_session.js` и `get_context.js`, добавить tool
`server_status` в `server.js`.

Последний шаг — обновить `registry.json` с полями `homelab_mcp_url` и
`homelab_mcp_enabled`.

## Критерии готовности

| Проверка | Как убедиться |
|---|---|
| Мост работает | `start_session` для LocOll содержит секцию `[Сервер с момента последней сессии]` |
| Graceful degradation | При `homelab_mcp_enabled: false` секция отсутствует, ошибок нет |
| Тайм-аут | При недоступном сервере `start_session` завершается не более чем через 3 секунды после основного ответа |
| `server_status` работает | Tool возвращает метрики железа + контейнеры + события |
| `server_status` без сессии | Работает без предварительного `start_session` |
| Нет крашей | Любая ошибка сети логируется в stderr и не роняет сервер |
